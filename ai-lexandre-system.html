System Architecture

AI-lexandre

Recruiter-facing interview assistant over my portfolio evidence.

AI-lexandre is a small production-style RAG system: a static GitHub Pages frontend, a Railway-hosted Fastify API, a generated local evidence corpus, role-aware retrieval, OpenAI answer generation, streaming UI updates, and citation filtering so the visible sources match the answer.

Architecture

The system separates the static portfolio experience from the interview runtime, so the public site stays fast while the API handles retrieval, model calls, and streaming.

Static Client

GitHub Pages

interview/index.html, app.js, and config.js render the chat UI, role presets, streaming tokens, sources, and follow-up chips.

API

Fastify on Railway

apps/interview-api exposes health, config, evidence search, full response, and streaming response endpoints.

Evidence

Generated Corpus

content.json is converted into chunks from projects, case studies, internships, education, and skills.

Model

OpenAI Responses

The API sends only the selected evidence blocks, role preset, question, and recent history to the model.

Request Flow

  1. The interview page loads API health and role/config data.
  2. The user picks a role preset and asks a question.
  3. The client calls /v1/interview/stream and falls back to /respond if streaming fails.
  4. The API validates the request, retrieves evidence, and builds a grounded model prompt.
  5. Tokens stream back to the UI, then the final payload updates answer text, sources, and follow-ups.

Evidence Flow

  1. content.json is the source of truth for public portfolio evidence.
  2. packages/interview-core builds a corpus with source type, title, section, citation label, URL, keywords, and role tags.
  3. Retrieval scores lexical matches, domain signals, role relevance, source type, and priority projects.
  4. Broad questions can diversify across all projects; domain questions prioritize direct domain evidence.
  5. The visible source chips are filtered to sources actually named in the final answer.

Runtime Contracts

The important behavior is encoded as small contracts between the client, API, retrieval package, and deployment environment.

Frontend

  • Reads the deployed API base URL from interview/config.js.
  • Maintains local chat state and selected role.
  • Parses NDJSON stream events: meta, token, done, and error.
  • Renders final sources from the completed payload, not from raw retrieval candidates.

API

  • Uses Zod schemas to validate requests.
  • Builds role-aware retrieval context before generation.
  • Streams through the OpenAI Responses API when available.
  • Falls back to deterministic grounded answers if provider generation fails before tokens are emitted.

Retrieval

  • Uses local structured portfolio content instead of scraping the live page.
  • Supports source types for projects, case studies, experience, education, skills, and overview text.
  • Boosts direct healthcare, hospital, patient, physician, and respiratory evidence for medical questions.
  • Diversifies results for broad inventory and healthcare-domain questions.

Deployment

  • GitHub Pages serves the portfolio and interview client.
  • Railway runs the Fastify API with OPENAI_API_KEY, OPENAI_MODEL, and CORS_ORIGIN.
  • MOCK_INTERVIEW_RESPONSES supports local deterministic mode.
  • The generated corpus is refreshed during the build path.

Architecture Audit

The current design is still appropriate for a portfolio-scale interview assistant, but the risk has moved from infrastructure to behavioral routing.

Right-sized

  • A static portfolio plus a small API is enough; a database, crawler, or vector store would add operational weight before the corpus needs it.
  • content.json as the source of truth keeps public pages and AI evidence aligned.
  • Lexical retrieval is transparent, cheap, and debuggable for a curated portfolio corpus.
  • Source-chip filtering is valuable because unsupported citations are more damaging than short answers.

Drift Risk

  • interview-service.ts now contains several intent branches for coursework, healthcare, quant fit, follow-ups, and source switching.
  • Those branches were added to fix real interview failures, but they can make the assistant feel scripted if the rules keep growing unchecked.
  • The deterministic fallback prose is intentionally simple and testable; the production OpenAI path should carry more natural wording.
  • The next refactor should be a small intent-policy table with scenario tests, not a larger retrieval stack.

Current Judgment

  • Not over-engineered at the system level: the deployment shape is modest and maintainable.
  • Slightly over-specialized in the service layer: behavior rules should stay visible and tested.
  • Not too simple for v1: the system has evidence coverage, compact memory, source alignment, fallback behavior, and API tests.
  • The main quality bar is no longer “can it answer,” but “does it answer like a natural interview while staying grounded.”

Why This Structure

  • It keeps the public portfolio static and cheap to serve, while the API can evolve independently.
  • It makes interview answers auditable because every answer starts from structured local evidence.
  • It lets role presets steer answers without hiding the actual retrieval basis.
  • It creates clear failure modes: API unavailable, streaming unavailable, provider unavailable, or weak evidence.
